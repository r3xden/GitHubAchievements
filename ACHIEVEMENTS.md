# ACHIEVEMENTS

*Every badge, in the order GitHub introduced it.*

</div>

## How to read this document

Each entry below is placed by the date GitHub introduced it, not by how hard it is to earn. A badge's **status** tells you whether it can still be unlocked today:

- 🟢 **Open** — earnable right now, by anyone, through ordinary use of GitHub.
- 🟡 **Limited** — technically earnable, but the path narrowed after GitHub changed a related feature.
- 🔒 **Closed** — tied to a one-time historical program. The window has passed; no amount of activity reopens it.

Full obtaining instructions live in [`OBTAIN.md`](./OBTAIN.md). This document is the map, not the route.

## I. The founding three (2021)

GitHub introduced the Achievements tab in April 2021, timed to the first flight of the Mars *Ingenuity* helicopter. Three badges launched together.

<br>

<div align="center">

### <img src="./assets/arctic-code-vault-contributor.png" width="28" valign="middle"> Arctic Code Vault Contributor

**Status: 🔒 Closed**

</div>

Awarded to anyone who had code merged into the default branch of an *active* public repository during GitHub's snapshot of the 2020 Archive Program — the one that etched the world's open-source code onto film and buried it in a decommissioned coal mine in Svalbard. The snapshot happened once. If your code was in a qualifying repository at that exact moment, you have this badge already; if it wasn't, no future contribution earns it. It's the closest thing GitHub has to a badge for *having been there.*

<br>

<div align="center">

### <img src="./assets/mars-2020-contributor.png" width="28" valign="middle"> Mars 2020 Contributor

**Status: 🔒 Closed**

</div>

Given to contributors of the specific open-source repositories that flew, quite literally, aboard the *Perseverance* rover and the *Ingenuity* helicopter — things like the F′ flight software framework. A narrow, specific badge: not "you contributed to space software" in general, but "you contributed to *this* software, in *this* mission." The qualifying repository list was fixed at launch and never grew.

<br>

<div align="center">

### <img src="./assets/public-sponsor.png" width="28" valign="middle"> Public Sponsor

**Status: 🟢 Open** *(originally "GitHub Sponsor")*

</div>

Earned by using GitHub Sponsors to financially back a developer or organization. Unlike its two siblings, this one was never tied to a moment in time — it's a standing recognition of putting money behind open-source work you rely on. It was renamed from "GitHub Sponsor" to "Public Sponsor" in mid-2022 alongside a design refresh, but the underlying criterion never changed.

## II. The 2022 expansion

In June 2022, GitHub added six more badges at once, each tied to an everyday collaboration pattern rather than a one-off program. This is where achievements stopped being commemorative and became something closer to a running record of how you work.

<br>

<div align="center">

### <img src="./assets/pull-shark.png" width="28" valign="middle"> Pull Shark

**Status: 🟢 Open** · Tiers: 2 → 16 → 128 → 1,024 merged PRs

</div>

The most common badge on GitHub, and deliberately so — it rewards the single most fundamental unit of collaboration: a pull request that got merged. It counts PRs merged into any repository, including your own, which makes it approachable for solo developers while still scaling meaningfully for maintainers who merge hundreds of contributions from others.

<br>

<div align="center">

### <img src="./assets/pair-extraordinaire.png" width="28" valign="middle"> Pair Extraordinaire

**Status: 🟢 Open** · Tiers: 1 → 10 → 24 → 48 co-authored merged PRs

</div>

Recognizes commits that carry a `Co-authored-by:` trailer inside a merged pull request — GitHub's native way of crediting pair programming or joint authorship. It's the one achievement that explicitly rewards *sharing* credit rather than accumulating it alone, which makes it a natural marker of collaborative habits rather than solo output.

<br>

<div align="center">

### <img src="./assets/quickdraw.png" width="28" valign="middle"> Quickdraw

**Status: 🟢 Open**

</div>

Unlocked by closing an issue or pull request within five minutes of opening it. It sounds like a badge for recklessness, but in practice it usually reflects the mundane reality of open-source maintenance: closing a duplicate issue, retracting a PR you opened by mistake, or tidying up a stale draft. Speed, not significance, is the criterion.

<br>

<div align="center">

### <img src="./assets/galaxy-brain.png" width="28" valign="middle"> Galaxy Brain

**Status: 🟡 Limited** · Tiers: 2 → 8 → 16 → 32 accepted answers

</div>

Earned when an answer you post in a GitHub Discussion's Q&A category gets marked "accepted" by the person who asked. In February 2024, GitHub disabled achievements inside its own Community Discussions forum to cut down on spam farming — so the badge no longer accrues there, but it still counts accepted answers in Discussions enabled on other public repositories. The path narrowed; it didn't close.

<br>

<div align="center">

### <img src="./assets/yolo.png" width="28" valign="middle"> YOLO

**Status: 🟢 Open**

</div>

Awarded for merging your own pull request without requesting or receiving a review. Small maintainers and solo repository owners earn this constantly and without trying, since there's often no one else around to review. On a larger team, it's a badge that quietly flags a moment where process was skipped — worth noticing, not necessarily worth repeating.

<br>

<div align="center">

### <img src="./assets/starstruck.png" width="28" valign="middle"> Starstruck

**Status: 🟢 Open** · Tiers: 16 → 128 → 512 → 4,096 stars

</div>

Given when a repository you created accumulates stars from other users. Of everything on this list, it's the least within your direct control — you can write good code and document it clearly, but you can't force anyone to star it. That's precisely why it tends to be read as a signal of genuine community value rather than personal effort alone.

## III. Reserved and internal

Not every badge shown by third-party trackers is something you can work toward. GitHub also uses the achievements *system* internally for its own staff, and has trialed badges publicly that were never fully released.

| Badge | Status | Note |
|---|:---:|---|
| Heart On Your Sleeve | 🔒 Closed / test | Trialed for ❤️ reactions on GitHub content; never left testing for general accounts. |
| Open Sourcerer | 🔒 Closed / test | Trialed for merged PRs across multiple public repositories; same fate as above. |
| Proxima Pioneer / Staffshipper / Staffuser | 🔒 Internal only | Awarded to GitHub employees for internal platform milestones. Not obtainable by external contributors under any circumstance. |

These are included for completeness, not as a checklist — they're documented here so you don't spend effort chasing something that was never open to begin with.
